# AGENTS.md — правила для агента

Это vault — персистентная память о проектах и задачах. Пользователь не редактирует файлы вручную: **все изменения вносишь ты, сразу по ходу разговора**. Полная методология — в `_rules.md` (прочитай при первой работе с vault).

Корень vault = корень этого репозитория.

## Главное

- Начало разговора о проекте → прочитай `projects/<проект>/<проект>.md`; тема неясна → `_dashboard.md`.
- Появился новый факт, решение, задача или статус → обнови соответствующий файл **немедленно**, не дожидаясь конца разговора.
- Задачи создавай из `_templates/task.md` со сквозным `id` (max по `projects/*/tasks/` + 1) и RICE-оценкой — детали в навыке `backlog`.
- Если vault — git-репозиторий: после содержательных правок `git add -A && git commit && git push`.

## ГЕЙТ ОТСЛЕЖИВАЕМОСТИ (обязательно)

**Прежде чем отчитаться пользователю о выполненной работе по проекту — убедись, что работа отражена задачей** (`todo` → `doing` → `review` → `done`). Нет задачи — заведи постфактум (в `review`, если готова к апруву; не в `done`). Это не опционально и не «в конце сессии»: проверяй на каждом ходе, где что-то сделал.

- **Что считается работой, требующей задачи:** новый код/фича, фикс бага, деплой, новый файл/страница/скилл/компонент, редизайн, миграция, правка конфигурации.
- **Это касается и инфра/мета-проектов** — самого́ vault, дашборда, скиллов, шаблонов. Тут перекос особенно частый: «продуктовым» проектам задачи заводятся, а работе над инструментами — забывается. Считается одинаково.
- **Не требует задачи:** чтение, исправление опечатки, ответ на вопрос без изменений в репозитории.
- Завершил задачу → навык `close`. **Потолок агента — `review`.** Закончил работу → переведи задачу в `review` + строку в лог сессии + предъяви отчёт (**#номер — название · что сделал · как проверил**). **`done` ставит только человеческий апрув** (через апрув-поверхность адоптера — напр. кнопку в дашборде). Агент в `done` не переводит: pre-commit-хук (`scripts/check-task-close.py`, `core.hooksPath .githooks`) **механически блокирует** агентский коммит с переходом в `done` (апрув-сервис обходит флагом `ARTEL_APPROVE=1`) — **при условии, что хук активирован** командой `git config core.hooksPath .githooks` (в свежем клоне автоматически не включается; см. `docs/installation.md`).
- **Закоммитить ≠ сделать.** Прежде чем нести на подтверждение, проверь, что работа реально работает (собралось, запустилось, страница открылась, в проде живо). Что проверить не смог — скажи прямо, не выдавай за проверенное. В отчёте укажи, *как* проверено.

## ХРАПОВИК ОШИБОК (обязательно)

Агент не учится между сессиями — переживают только файлы и механические гейты, а прозовые правила нарушаются. Поэтому «обучение на ошибках» здесь — свойство системы, не обещание агента.

- **Повторная ошибка ⇒ механический гейт, не напоминание.** Первый раз — можно правило/запись. Повтор класса — обязан механизм (хук, валидатор, структурное изменение), работающий без участия агента.
- Значимый инцидент → запись в `_lessons.md` (*что · почему · механизм или честно «дыра + план» · класс*). `_lessons.md` читается в начале сессии (`sync`).
- Механизм поставлен, а ошибка класса повторилась → чинить **механизм**, не обещать «постараюсь».
- **Открытые запросы:** многошаговую/отложенную просьбу пользователя сразу заноси в `_requests.md` (`- [ ]`), отмечай `[x]` по выполнении. Навык `sync` показывает незакрытые в начале сессии; если ваш агент-адаптер поддерживает хуки (напр. Claude Code — UserPromptSubmit), переинъекцию незакрытых перед каждым ходом стоит автоматизировать — чтобы запрос не выпал из внимания за длинную работу.

Формат и пример — `docs/lessons.md`. Ценность фреймворка для адоптера: не «ИИ умнеет», а «система не откатывается по уже пойманной ошибке».

## АРТЕЛЬ — команда ролей и документация (обязательно)

Полная методология — в `docs/` (`roles`, `workflow`, `doc-canon`, `styleguide`). Кратко:

- **Роли по RICE.** При заведении задачи проставляй `roles` (навык `backlog`): `reviewer` всегда; `analyst` при `confidence ≤ 60`; `techwriter` при `reach ≥ 6` или изменении архитектуры. Роли действуют через чтение `docs/roles.md` — исполняй их мандат.
- **Оценка в Story Points.** Размер задачи — `sp` (Фибоначчи 1/2/3/5/8/13), не время: у людей валюта — время, у ИИ — токены. `rice_effort = sp/5`; `sp ≥ 8` → декомпозируй. Старые задачи с `est_days` не трогаются.
- **Грейд модели.** Проставляй `model_tier` (навык `backlog`) по размеру задачи `sp`: `junior` (`sp ≤ 2`), `middle` (`sp` 3–5), `senior` (`sp ≥ 8`). Грейды вендоронейтральны (маппинг на модели — в адаптере); эскалация на ступень вверх при затыке ловит рассуждательную сложность мелких задач. Метрика стоимости SP = факт-токены / `sp`. Методология — `docs/methodology.md` (Тиринг моделей).
- **Гейт «готово».** Перед тем как нести на подтверждение — пройди ревью: корректность (баги, безопасность) **и полнота** (что упущено, что не проверено, какой документ устарел).
- **Документация — обязательный артефакт.** Канон по зрелости проекта (`docs/doc-canon.md`); документ существует только при ясных Audience и Purpose (APC, `docs/styleguide.md`); без воды. Изменение касается пользователя → строка в `CHANGELOG` (навык `close`). Дока **обязана соответствовать реальности**.
- **Перед деплоем — прочитай runbook проекта**, не реверс-инжинирь. Нет runbook → сначала задокументируй. Деплой-конфиг живёт в репо, не в локальном игноре.
- **Релиз-гейт (все проекты):** доки и релизноутс актуальны, отложенное заведено задачами, в конфигах нет личных данных.

## Навыки

- `sync` — загрузить контекст всех проектов в сессию
- `backlog` — завести задачу (RICE, номер тикета, роли)
- `close` — закрыть задачу (done + лог сессии); сюда же — постфактум-задача на уже сделанную работу
- `recap` — собрать итоги за период (день/неделя/месяц) для ревью и саморефлексии; когда пользователь прощается — дай итоги дня сам
- `inbox` — разобрать `_inbox/` в задачи

## Чего не делать

- Не удалять информацию без явной просьбы пользователя
- Не менять структуру папок и заголовков
- Не задавать в чате вопросы по задаче — записывай их в раздел «Вопросы» файла задачи
