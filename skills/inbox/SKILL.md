# inbox

Разобрать инбокс: каждую сырую запись из `_inbox/` превратить в полноценную задачу (или идею/выполненное действие) и удалить запись. Самодостаточно: можно дать любому агенту.

## Шаг 0 — прочитай правила

Прочитай `_rules.md` в корне vault. Создание задач здесь идёт по тому же контракту, что и в навыке `backlog` (см. ниже краткую выжимку; если доступен файл `backlog/SKILL.md` — свериться с ним).

## Контракт (выжимка из backlog)

Задача → `projects/<slug>/tasks/<имя>.md`. Frontmatter: `id` (максимальный по vault + 1), `project: "[[slug/slug]]"`, `status: todo`, `tags: [task]` (+`bug` для багов), `created`/`updated` = сегодня, `sp` (Story Points — Фибоначчи 1/2/3/5/8/13), `rice_reach` (1–10), `rice_impact` (1–5), `rice_confidence` (50–100), `rice_effort` (= sp/5, min 0.1), `summary`, `roles`, `model_tier`. Разделы тела: Что нужно сделать · Почему важно · Критерии готовности (DoD, наблюдаемый результат) · Пререквизиты · Вопросы (в файл, не в чат) · Заметки (сюда же план реализации, если нужен). DoD-чеклист = прогресс. Содержимое — на русском.

Формат записи инбокса (`_inbox/<любое-имя>.md`):

```yaml
---
created: <ISO-дата>
project: <slug или отсутствует>
tags: [inbox]
---

<текст мысли>
```

## Шаги

1. **Подтянуть свежее.** Если vault — git-репо: `git pull --rebase --autostash` (записи могут приходить с других устройств).
2. **Собрать записи.** Все `_inbox/*.md`, кроме `.gitkeep`. Пусто → сказать пользователю «инбокс пуст» и закончить.
3. **Для каждой записи определить тип и обработать:**
   - **Задача** («сделать X», «починить Y», «добавить Z») → создать файл задачи по контракту выше. Проект из frontmatter `project`; нет — определить из текста; сомнение → в «Вопросы» задачи (не в чат). Оригинальный текст сохранить в «Заметках»: `Из инбокса <created>: "<текст>"`.
   - **Идея/заметка на потом** («подумать про…», ссылка, наблюдение) → не плодить мусорную задачу: вынести в карточку проекта (раздел про идеи) или в `roadmap.md` («Идеи на будущее»).
   - **Поручение проверить/выяснить** («проверь, сделана ли X», «кмк Y готово») → по возможности **выполнить сразу** (проверить факты по репозиторию/файлам), зафиксировать результат (обновить статус нужной задачи, при необходимости завести новую), и отчитаться, что выяснилось.
4. **Удалить разобранный файл** из `_inbox/`.
5. **Commit + push** (если git): `git add -A && git commit && git push`.
6. **Отчитаться** списком: для каждой записи — что создано/сделано (проект, файл, RICE) и где были сомнения.

## Пример (запись → результат)

**Запись** `_inbox/2026-06-12-0913.md`:
```
---
created: 2026-06-12T09:13:00.000Z
project: myapp
tags: [inbox]
---
кнопка экспорта на мобиле наезжает на заголовок
```

**Действие:** это баг → создать `projects/myapp/tasks/fix-export-button-overlap.md` с `tags: [task, bug]`, `summary: "🐛 Кнопка экспорта наезжает на заголовок на мобиле"`, оценкой RICE (узкое визуальное, но видно всем мобильным: reach 5, impact 2, confidence 80, sp 1 → effort 0.2 → RICE ≈ 40), DoD «на мобильной ширине кнопка экспорта не перекрывает заголовок», в «Заметках» — оригинальный текст. Затем удалить запись, закоммитить.

## Важные правила

- **Не оставлять инбокс наполовину разобранным.** Каждая запись либо стала задачей/идеей и удалена, либо явно пропущена с объяснением пользователю.
- **Не редактировать `_inbox/` в обход цикла** (write задачу → delete запись → commit → push), чтобы не конфликтовать с устройствами, пишущими в инбокс.
- Сомнения — в «Вопросы» файла задачи, не в чат.
- Содержимое задач — на русском.

## Самопроверка перед завершением

- [ ] Каждая запись обработана: задача / идея / выполненное поручение
- [ ] Все новые задачи прошли контракт (уникальный id, числовой RICE, DoD-результат)
- [ ] Все разобранные файлы удалены из `_inbox/`
- [ ] (если git) запушено
- [ ] Пользователю выдан понятный отчёт по каждой записи
