# Локальный дашборд

Веб-витрина vault без зависимостей и без сборки. Один Node-скрипт читает markdown-файлы и отдаёт страницы: проекты, канбан (todo / в работе / на ревью / готово), беклог по RICE, инбокс, аналитика, поиск, справка (`/help`), страницы задачи (`/t/<id>`) и проекта (`/p/<slug>`) по клику.

> **Это намеренно минимальная, демонстрационная версия.** Один файл, без зависимостей и сборки — чтобы любой пользователь шаблона сразу что-то увидел. Она показывает суть методологии (проекты · канбан · RICE · инбокс · аналитика · поиск), но это **не полноценный продукт**: нет Ганта/роадмапа, диаграммы сгорания и хостинга. Запись ограничена двумя действиями — быстрый захват в инбокс и кнопка апрува `review → done`; всё остальное (задачи, статусы) пишет агент в markdown.
>
> Структура vault — обычный markdown с frontmatter, так что поверх неё несложно собрать полноценное приложение (например, на Next.js) с этими возможностями. Этот скрипт — точка старта и референс, а не потолок.

## Запуск

Из корня vault:

```bash
npm run dashboard          # откроет http://localhost:4321 в браузере
# или напрямую:
node dashboard/serve.js              # без авто-открытия
node dashboard/serve.js --port 8080  # другой порт
node dashboard/serve.js --open       # открыть в браузере
```

Нужен только Node.js 18+ (никаких `npm install`).

## Как это работает

- Данные читаются из файлов **при каждом запросе** — правки агента видны по обновлению страницы (страница сама перезагружается каждые 2 минуты).
- Парсятся: карточки проектов (`projects/*/<slug>.md` с тегом `project`), задачи (`tasks/*.md` с тегом `task`), записи инбокса (`_inbox/*.md`).
- RICE считается на лету: `(reach × impact × confidence%) / effort`.
- Канбан/RICE: грейд-бейджи 🐣 junior / 🦆 middle / 🦅 senior (из `model_tier`), фильтры по проекту и грейду (без перезагрузки), счётчики «+N / ✓N сегодня». Клавиша `/` фокусирует поиск.
- Аналитика: плановые грейды (по сложности тикетов из `sp`) **и** фактическое использование моделей по грейдам (из `cost_by_model`, сгруппировано через маппинг адаптера «Тиринг — адаптер» в `AGENTS.md`). Если адаптер не объявил маппинг — карточка подсказывает его задать.
- Записи две, остальное — чтение:
  - **Захват в инбокс** — форма на вкладке «Инбокс» (`POST /api/inbox`) создаёт markdown в `_inbox/` (формат навыка `inbox`).
  - **Апрув** — кнопка «Принять» на задачах в статусе `review` (`POST /api/approve`) проставляет `status: done` + `closed_at`. Это и есть человеческая апрув-поверхность из методологии (`done` ставит человек, не агент).
  - Git по умолчанию не трогается — файлы разбирает/коммитит агент. Если задан env `ARTEL_GIT_PUSH`, апрув коммитит+пушит с `ARTEL_APPROVE=1` (обход pre-commit-гейта).
  - ⚠️ Без `ARTEL_GIT_PUSH` апрув пишет `done` только в файл. Следующий коммит этого перехода агент должен сделать с `ARTEL_APPROVE=1` — иначе pre-commit-гейт его заблокирует (так и задумано: апрув — полномочие человека).
- Авто-перезагрузка страницы (каждые 2 минуты) ставится на паузу, пока вы печатаете в поле инбокса или поиска, чтобы не стереть ввод.
- Закрыли терминал → дашборд остановился.

## Запустить на другом vault

```bash
node dashboard/serve.js --root /путь/к/другому/vault
```

## Чем отличается от Obsidian-сводки

`_dashboard.md` + Dataview работают внутри Obsidian. Этот дашборд — отдельная веб-страница: удобно держать открытой во вкладке, не требует Obsidian, видна с телефона в той же сети (`http://<ip-компа>:4321`).
