# Методология

Зачем это всё: LLM-агент между сессиями ничего не помнит, а человек не хочет вести трекер руками. Vault решает обе проблемы — это **внешняя память агента**, которую он сам и поддерживает, и одновременно читаемый человеком трекер. Три принципа:

1. **Агент пишет, человек читает.** Вы разговариваете — агент заводит задачи, двигает статусы, ведёт лог. Руками файлы не редактируются (можно, но незачем).
2. **Отслеживаемость.** Каждое значимое действие агента отражено в задаче. По vault всегда можно восстановить, что делалось, когда и почему.
3. **Обновлять сразу.** Не «в конце сессии», а в момент, когда появился факт, решение или задача. Сессия может оборваться — vault уже актуален.

## Проект

`projects/<slug>/<slug>.md` — карточка: суть, ЦА и боль, видение, метрики, топ-задача, **лог сессий**. Лог — секции «Что сделано в сессии \<дата\>» сверху вниз от новых к старым: это хроника проекта, по которой новый агент (или вы через месяц) восстанавливает контекст за минуту.

Frontmatter:

| Поле | Значения |
|---|---|
| `status` | `idea` / `active` / `paused` / `done` |
| `type` | `product` / `tool` / `research` / … (свободно) |
| `priority` | `high` / `medium` / `low` |
| `audience` | для кого |
| `repo`, `local` | ссылка на код и локальный путь (если есть) |
| `updated` | дата последнего изменения |

`roadmap.md` — фазы с чекбоксами. Не план-обещание, а живой документ: выполненное отмечается, новое дописывается.

## Задача

`projects/<slug>/tasks/<имя>.md`. Имя файла — короткое, на английском, kebab-case (`csv-export.md`).

Frontmatter:

| Поле | Что это |
|---|---|
| `id` | **Сквозной номер тикета** по всему vault: max существующего `id` + 1. Никогда не переиспользуется |
| `status` | `todo → doing → done`, плюс `blocked` и `cancelled`. **`blocked`** — задачу нельзя взять сейчас по любой причине (ждёт внешнего/доступа/окружения, опасная операция в текущих условиях, зависит от незакрытой задачи); блокер пишется в задаче. Не висит в `doing` |
| `created` / `updated` | даты; `updated` трогается при каждом изменении |
| `sp` | размер задачи в **Story Points** (Фибоначчи 1/2/3/5/8/13), см. RICE-раздел. Не время |
| `rice_*` | компоненты приоритета, см. ниже |
| `summary` | одна строка для индексов и дашборда; можно с эмодзи-маркером (🐛 баг, 🚨 риск) |
| `tags` | `[task]`; для багов `[task, bug]` |

Разделы тела — и чем они отличаются:

- **Что нужно сделать** — описание задачи.
- **Почему важно** — связь с целью проекта (если её нет, задача — кандидат на cancelled).
- **Критерии готовности (DoD)** — *наблюдаемые условия результата*: что станет правдой, когда задача сделана. «Страница /x открывается после авторизации», а не «сверстать страницу». Неупорядоченный чеклист — и он же мера прогресса: отмеченные галочки показывают, насколько задача готова.
- **Пререквизиты** — зависимости и блокеры (или «нет»).
- **Вопросы** — что прояснить до старта. Агент пишет вопросы **в файл**, а не задаёт в чате — чтобы не блокировать разговор и не терять их.
- **Заметки** — происхождение («Из инбокса …»), контекст, ссылки. Сюда же — план реализации, если он нужен (отдельной секции «План» нет: план — это «как делать», внутренняя кухня исполнителя, а пользователю важны DoD и прогресс по нему).

## RICE-приоритизация

```
RICE = (reach × impact × confidence%) / effort
```

| Компонент | Шкала | Как оценивать |
|---|---|---|
| `rice_reach` | 1–10 | Как часто/широко эффект встречается. Ежедневная боль — 8–10, редкий кейс — 1–2 |
| `rice_impact` | 1–5 | Сила эффекта на цель проекта. Критичный риск или ключевая метрика — 5, косметика — 1 |
| `rice_confidence` | 50–100 | Уверенность в оценках выше. Проверено/очевидно — 90–100, гипотеза — 50–70 |
| `rice_effort` | — | **Выводится**: `sp / 5`, минимум 0.1. Не оценивается отдельно |

Размер задачи оценивается в **Story Points** (`sp`, Фибоначчи 1/2/3/5/8/13) — не во времени: у людей валюта — время, у ИИ-агента — токены, поэтому первичная оценка отвязана от часов (velocity ИИ = токены / SP, см. «Тиринг моделей»). `sp ≥ 8` — сигнал декомпозировать. Якоря шкалы — в навыке `backlog`. Существующие задачи с `est_days` не мигрируются (переходный период).

Беклог сортируется по RICE автоматически (Dataview) — «что брать следующим» перестаёт быть спором с собой. Оценки грубые, и это нормально: их задача — отделить RICE 10 от RICE 0.5, а не спорить о 3.2 против 3.6.

Для `done`/`cancelled` RICE в индексах показывается как «—»: приоритет мёртвой задачи не имеет смысла.

## Жизненный цикл

```
мысль → _inbox/ → (навык inbox) → задача todo
                                      ↓ взяли в работу: status doing
                                      ↓ сделали: status done, лог сессии в карточке проекта
обсудили будущую фичу  → задача todo (сразу, из разговора)
сделали что-то без задачи → задача создаётся постфактум со статусом review (агент не ставит done)
передумали → cancelled (не удалять: история решений тоже память)
```

Правило, которое держит систему честной: **нельзя закончить работу в разговоре, не обновив статус задачи**. Отдельный принцип: **`doing` ставится до старта, не после** — иначе задача невидима на борде, пока ведётся работа. Прыжок `todo → review/done` в обход `doing` — частная форма нарушения (детально и гейты — в [workflow.md](workflow.md)).

## Инбокс

`_inbox/` — буфер для сырых мыслей, когда некогда формулировать: по одному markdown-файлу на запись (формат — в [_rules.md](../_rules.md)). Записи может класть что угодно: веб-форма, iOS Shortcut, телеграм-бот, `echo > файл`.

Разбор (навык `inbox`): каждая запись становится полноценной задачей с RICE (оригинальный текст сохраняется в «Заметках»), не-задачи уходят в карточку проекта или роадмап, поручения по возможности выполняются сразу. Файл записи удаляется. Инбокс никогда не остаётся разобранным наполовину.

## Команда ролей и документация

Беклог выше отвечает, *что* делать и *в каком порядке*. Дальше — *как* над задачей
работает команда. Роли и канон действуют, как только агент читает эти документы:
настройки и конкретного вендора не требуют.

- **[Роли](roles.md)** — аналитик, разработчик, ревьюер, техписатель. Подключаются
  по компонентам RICE задачи: мутная (низкий confidence) → аналитик; высокий impact
  → ревьюер углубляется; широкий reach → техписатель.
- **[Жизненный цикл и гейты](workflow.md)** — где роли включаются, гейт «готово»,
  релиз-гейт. Подробная версия цикла, описанного выше.
- **[Документация](doc-canon.md)** — обязательный канон внешней и внутренней доки по
  зрелости проекта, по авторской модели **APC** (Audience · Purpose · Collaboration).
- **[Стайлгайд](styleguide.md)** — как писать: APC, без воды, формат под агента.

## Тиринг моделей

Зависимость от одной топ-модели — точка отказа (бан, недоступность, цена). Артель
снижает риск: задаче назначается **грейд модели**, и по умолчанию работает самая
дешёвая, что справится. Это вторая опора вендор-устойчивости рядом с
вендор-нейтральной докой — чем полнее `AGENTS.md`, тем больше задач выполнимо на
младшем грейде.

Плановый грейд выводится из **размера задачи (`sp`)**, не из её важности по RICE
(важная задача бывает мелкой, неважная — крупной). Размер — рабочий прокси
сложности: крупной задаче нужен более сильный исполнитель. Рассуждательную
сложность мелкой задачи (низкий `confidence`, смена архитектуры) ловит эскалация —
она поднимает грейд выше планового по факту.

### Три грейда

Грейды названы по уровню исполнителя — вендоронейтрально. Конкретная модель за
каждым грейдом задаётся в **адаптере проекта/инструмента**, не в каноне (как
`AGENTS.md` → `CLAUDE.md`): у одного вендора это junior=быстрая, middle=рабочая,
senior=рассуждающая; вендор с двумя моделями схлопывает соседние грейды.

| Грейд | Размер | Когда |
|---|---|---|
| **junior** | `sp ≤ 2` | мелкие, механические |
| **middle** | `sp` 3–5 | средние (дефолт) |
| **senior** | `sp ≥ 8` | крупные, сложные |

Плановый грейд пишется в `model_tier:` frontmatter — выводится при заведении
(навык `backlog`) из `sp`, как и `roles`. Границы `sp` стартовые,
**самокалибруются по доле эскалаций**.

### Эскалация — «дёшево, но не любой ценой»

Плановый грейд берётся по умолчанию; при затыке или низкой уверенности в результате
задача поднимается на ступень вверх (`junior → middle → senior`) — как стажёр
поднимает руку. По завершении фиксируется **фактический грейд** (с учётом
эскалаций) — он, а не плановый, идёт в метрики.

### Механика (гибрид)

Грейд — рекомендация, исполнение зависит от того, делегируема ли работа:

- **Делегируемое** — оркестратор запускает субагента нужного грейда и эскалирует
  вверх при затыке.
- **Неделегируемое** (основной цикл агента) — грейд показывается человеку, он
  ставит модель сессии под задачу.

Канон описывает грейды и правило вывода; *какой моделью* их закрывать — дело
адаптера конкретного инструмента.

### Пример

Задача «вынести формулу дневного лимита в утилиту» — `sp 1`, `model_tier: junior`.

- **Делегируемо.** Оркестратор (на senior-модели) видит грейд и запускает
  субагента на junior-модели: даёт задачу, получает результат, прогоняет через
  гейт «готово». В Claude Code это субагент с моделью junior-грейда (Haiku).
  Junior «взял задачу» = оркестратор её делегировал. Забуксовал → эскалация на middle.
- **Неделегируемо** (задачу ведёт человек в диалоге). Автопереключения нет —
  модель сессии фиксирована. Грейд `junior` — сигнал человеку: можно вести на
  дешёвой модели → он ставит её на сессию (`/model` в Claude Code) или открывает
  отдельную дешёвую сессию.

Тег `model_tier` **маршрутизирует, но не запускает**: автономного диспетчера
«беклог сам раздаёт задачи моделям» нет — исполнение инициирует оркестратор
(делегируемое) или человек.

### Мандат грейда (адаптер)

Адаптер задаёт не только маппинг грейд→модель, но и **поведение исполнителя**.
Дешёвую модель можно запустить самостоятельным агентом с мандатом: *«ты junior —
берёшь из беклога только `model_tier: junior`, выполняешь по DoD; задачи выше
грейда не трогаешь, при затыке поднимаешь руку (эскалация)»*. Тогда junior-агент
(например, Haiku) сам фильтрует беклог по своему грейду и работает автономно в его
пределах, а оркестратор/человек подключается на эскалациях и приёмке. Маппинг и
мандат — в адаптере (`AGENTS.md`-слой проекта), не в каноне.

### Контракт адаптера (обязательно)

Грейд работает, только если адаптер инструмента объявил, *чем* его закрывать. Это
**обязательство адаптера**: без него `model_tier` — мёртвая метка (проставляется,
но никуда не маршрутизирует). Канон конкретные модели сюда не вписывает — это и есть
точка вендоронейтральности; но он требует, чтобы адаптер объявил их у себя. Место
объявления — **секция в `AGENTS.md`-слое адаптера** (для Claude Code — через
`CLAUDE.md → @AGENTS.md`); нулевая инфраструктура.

Объявление **самообслуживаемое**: маппинг заполняет сам подключённый агент при первом
заходе — он знает свои модели и умеет ли делегировать. Мейнтейнеры Артели не пишут
маппинг за каждого вендора; это не масштабируется и противоречит вендоронейтральности.
Канон даёт слот и инструкцию (`AGENTS.md` → «Тиринг — адаптер»), агент заполняет его
под себя. Так новый вендор включает тиринг сам, без правок ядра.

Секция адаптера обязана задать четыре вещи:

1. **Маппинг грейд→модель** — junior/middle/senior → конкретные модели вендора.
   Вендор с двумя моделями схлопывает соседние грейды; с одной — тиринг вырождается
   в заглушку (допустимо, но объяви явно).
2. **Механику исполнения** — как грейд приводит к запуску: есть субагенты →
   делегирование (оркестратор запускает субагента нужного грейда); нет субагентов →
   грейд показывается человеку, он ставит модель сессии. Автономного диспетчера канон
   не требует.
3. **Эскалацию** — как грейд поднимается при затыке (`junior→middle→senior`) и где
   фиксируется фактический грейд.
4. **Запись токенов** — как адаптер пишет `cost_by_model` (io-токены раздельно по
   моделям) на задачу из usage сессии. Без записи метрика «Метрики» (токены/SP по
   моделям и грейдам, ниже) не вычислима — остаётся пустой. Адаптер, который не умеет
   читать usage, обязан объявить это явно: метрика тогда честно недоступна, а не молча
   пуста. Цепочка «считай → **записывай** → собирай в аналитику» рвётся именно на записи,
   если её не сделать обязанностью адаптера.

Шаблон секции (скопируй в `AGENTS.md` адаптера и заполни):

```
## Тиринг — адаптер <вендор/инструмент>
- Маппинг: junior=<модель>, middle=<модель>, senior=<модель>
- Механика: <делегирование субагентами | человек ставит модель сессии>
- Эскалация: <как поднимается грейд при затыке; где фиксируется факт-грейд>
- Запись токенов: <как пишется cost_by_model на задачу | «не умею» — метрика недоступна>
```

Референс — Claude-адаптер (llmush): junior=Haiku (субагент `Agent model:haiku`),
middle=Sonnet, senior=основной цикл (Opus); механика — делегирование субагентами +
гейт делегирования; эскалация `junior→middle→senior`, факт-грейд из `cost_by_model`;
запись токенов — `scripts/task-checkpoint.py` (снимает usage из транскрипта) +
`SubagentStop`-хук (токены субагентов по ключу тикета). Не объявил секцию — грейды в
беклоге проставляются, но тиринг не включён.

### Метрики (на данных, отдельная итерация)

Метрика ведётся **в токенах, раздельно по моделям** — не в деньгах: работа обычно
идёт по подписке (фикс/мес), а не по API-оплате за токен, поэтому $ по API-прайсу —
фикция; и токен senior ≠ токен junior, складывать их нельзя. Знаменатель —
**`sp`** (нормализует на размер задачи). Это velocity ИИ: токены на единицу размера,
аналог «часов на SP» у людей. Источник факта — транскрипты сессии (`usage` + `model`);
io-токены (input+output) = чистая работа, `cache_read` ведётся отдельно (шум
накопленного контекста, растёт к концу сессии). $ — опциональная справка через
прайс-адаптер, не основная метрика.

- **токенов на 1 SP** = io-токены (по модели) / `sp`, в разрезе грейда и проекта.
  `sp` — плановая грубая оценка, поэтому метрика осмысленна на агрегатах, не
  на единичной задаче; расхождение план/факт само диагностично (дорогой SP →
  задача переоценена или проект сложный);
- **доля эскалаций** — диагностика: где дока или постановка недотянули;
- **доля задач на junior** — прямой измеритель вендор-независимости, растёт по мере
  улучшения документации.

Твист для ИИ: модель не «учится» в проекте между задачами — её онбординг это и есть
документация. Поэтому стоимость SP по проекту — измеритель качества доки для агента:
падает → дока работает.
