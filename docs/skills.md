# Навыки

Навыки лежат в `skills/` **внутри vault** и едут с каждым клоном. Пути в них — относительно корня репозитория, ничего настраивать не нужно. Это markdown-инструкции для агента — любой агент исполняет их, прочитав по указанию из `AGENTS.md`; Claude Code подхватывает их как `/`-команды автоматически.

Каждый навык — это `SKILL.md`: инструкция для агента, когда и как действовать. Никакого кода — логику исполняет сам агент по шагам из файла.

## sync — загрузить контекст

**Вызов:** `/sync`, или словами — «sync», «что по проектам», «обзор».

Читает `_rules.md`, `_dashboard.md` и все карточки проектов с задачами, отдаёт сводку: проекты со статусами, активные задачи по убыванию RICE, недавно завершённые. По умолчанию read-only; при наличии запланированных (`next_up: true`) задач берёт верхнюю в работу (`doing`).

**Когда полезен:** в начале сессии, чтобы агент вошёл в курс дела; раз в несколько дней как «планёрка с собой».

## backlog — завести задачу

**Вызов:** `/backlog <описание>`, или «добавь в беклог: …», «заведи задачу …».

Определяет проект, назначает сквозной `id`, оценивает `sp` (Story Points) и RICE, создаёт файл задачи со всеми разделами по шаблону, коммитит. Сомнения и открытые вопросы пишет в раздел «Вопросы» файла — не спрашивает в чате.

**Когда полезен:** мысль появилась посреди другой работы — одна фраза, и она в беклоге с приоритетом, а вы не потеряли фокус.

## close — закрыть задачу

**Вызов:** `/close`, или «закрой задачу», «готово» — а ещё агент применяет его сам, когда что-то доделал.

Готовит задачу к закрытию: проверяет, что работа реально работает, предъявляет пользователю что сделано по тикету и **ждёт подтверждения**. `done` ставится только после явного ока (+ строка в лог сессии). Если задачи не было — заводит постфактум в `review`. Это закрывает главный провал отслеживаемости: работу над инструментами (дашборд, сам vault, скиллы) легко сделать и забыть залогировать. Гейт в `AGENTS.md`/`_rules.md` прямо запрещает отчитываться о готовом, пока нет задачи.

## recap — итоги за период

**Вызов:** `/recap`, или «итоги дня», «итоги недели/месяца», «что я сделал за…», «для ревью».

Собирает читаемые итоги: хайлайты + результаты по проектам (в терминах outcome, не дамп задач), из закрытых задач за период и логов сессий. Для 360-ревью, отчёта руководителю и саморефлексии — когда сделанное теряется в потоке. Read-only; по запросу «сохрани» пишет в `recaps/<период>.md`.

## inbox — разобрать инбокс

**Вызов:** `/inbox`, или «разбери инбокс».

`git pull --rebase --autostash` → каждая запись `_inbox/*.md` становится задачей (или идеей в карточке проекта, или выполняется сразу, если это поручение) → запись удаляется → commit + push → отчёт списком.

**Когда полезен:** раз в день-два, если вы кидаете мысли в инбокс с телефона или другого устройства.

## Как добавить свой навык

1. Создайте `skills/<имя>/SKILL.md`.
2. Frontmatter:
   ```yaml
   ---
   name: имя
   description: Что делает и когда использовать (агент выбирает навык по этому описанию).
   user_invocable: true
   ---
   ```
3. Тело — пошаговая инструкция: что прочитать, что создать/изменить, что ответить пользователю, правила и ограничения. Пишите как ТЗ исполнителю.
4. Пути указывайте от корня репозитория — навык останется переносимым.

Идеи навыков, которые напрашиваются: «недельный обзор» (что сделано за неделю по всем проектам), «гигиена беклога» (протухшие doing, задачи без RICE).

## Перенос в свою систему промптов

`SKILL.md` — обычный markdown: любая модель, умеющая работать с файлами, может исполнить те же шаги — просто покажите ей файл навыка или перенесите его содержимое в свою систему промптов. Конвенции vault от инструмента не зависят.
