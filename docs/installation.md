# Установка и настройка

## Вариант 1: новый vault из шаблона (рекомендуется)

1. На странице [github.com/kringova/artelush](https://github.com/kringova/artelush) нажмите **Use this template → Create a new repository**. Сделайте свой репозиторий **приватным** — в vault будет жить контекст ваших проектов.
2. Склонируйте к себе:
   ```bash
   git clone git@github.com:<вы>/<ваш-vault>.git ~/vault
   ```
3. Откройте папку в Claude Code:
   ```bash
   cd ~/vault && claude
   ```
   Правила подхватятся из `CLAUDE.md` (он импортирует канонический `AGENTS.md`), навыки — из `skills/`.
4. Проверьте: скажите агенту «заведи проект тест» — он должен создать `projects/test/` со структурой по шаблону. Потом «удали проект тест и example-project».

## Вариант 2: без GitHub

Просто склонируйте шаблон и удалите привязку к origin:

```bash
git clone https://github.com/kringova/artelush.git ~/vault
cd ~/vault && git remote remove origin
```

Git-синхронизация опциональна: vault работает и как обычная папка. Но с git вы бесплатно получаете историю изменений и работу с нескольких устройств.

## Вариант 3: встроить в существующий код-репозиторий

Если не нужен центральный vault, а нужен трекер внутри одного проекта:

1. Скопируйте в корень репозитория: `_rules.md`, `_templates/`, `_inbox/`, папку `skills/` (или слейте с существующей).
2. Создайте `projects/<имя-проекта>/` по образцу `example-project`.
3. В свой `CLAUDE.md` добавьте `@AGENTS.md` (или скопируйте `AGENTS.md` шаблона целиком). Инструменты, читающие `AGENTS.md` сами (Cursor, Codex и др.), подхватят его без шима.

Навыки ищут vault от корня репозитория, поэтому работают и во встроенном режиме.

## Obsidian (опционально)

Vault — обычный markdown, Obsidian не обязателен. Если хотите визуальные сводки:

1. Obsidian → Open folder as vault → папка vault.
2. Установите community-плагин **Dataview** и включите его.
3. Откройте `_dashboard.md` — таблица проектов, беклог по RICE, канбан. В каждом `projects/X/tasks/tasks.md` — индекс задач проекта.

Wiki-ссылки (`[[...]]`) кликабельны в Obsidian; для агента это просто текст с понятной семантикой.

## Несколько устройств

Работает через git. Единственное правило: перед разбором инбокса и в начале сессии на «другом» устройстве — `git pull --rebase --autostash` (навык `inbox` делает это сам). Конфликтов почти не бывает: записи инбокса — отдельные файлы, задачи редактируются по одной.

## Гейт закрытия (review → done)

Артель механически блокирует агентский переход задачи в `done`: агент не может закоммитить задачу со статусом `done` без явного сигнала от апрув-сервиса. Потолок агента — `review`; `done` ставит только человек через апрув-поверхность адоптера (например, кнопку в дашборде).

Реализовано через git pre-commit хук (`scripts/check-task-close.py` + `.githooks/pre-commit`).

### Установка

1. Активируйте хук:
   ```bash
   git config core.hooksPath .githooks
   chmod +x .githooks/pre-commit scripts/check-task-close.py
   ```
2. Проверьте: попробуйте закоммитить задачу с `status: done` без флага — хук должен заблокировать коммит с сообщением об ошибке.
3. Проверьте апрув-поверхность: выполните `POST /api/approve` на задаче с непустым полем `sp` и убедитесь, что `sp` и остальные frontmatter-поля сохранились без изменений. В прошлом был баг regex, который затирал поля при апруве — проверьте на своём адаптере перед использованием в prod.

### Апрув-флоу

Когда апрув-поверхность адоптера (дашборд, скрипт, CI-шаг) подтверждает закрытие, она выставляет `ARTEL_APPROVE=1` перед коммитом — хук пропускает:

```bash
ARTEL_APPROVE=1 git commit -m "close: task-123 done"
```

Агент в `done` не переводит: даже если он напишет `status: done`, хук заблокирует коммит. Обойти можно только явно выставив флаг — это намеренная улика, а не дыра.

## Переезд на новый компьютер

`git clone` + открыть в Claude Code. Всё, что нужно агенту (правила, шаблоны, навыки), едет внутри репозитория — глобальная настройка не требуется.

## Чеклист навыков

После клонирования или ручного синка шаблона убедитесь, что vault содержит все обязательные навыки. При ручном переносе файлов навык легко потерять.

Обязательные навыки в `skills/`:

- [ ] `sync` — загрузить контекст проектов
- [ ] `backlog` — завести задачу
- [ ] `close` — закрыть задачу / отправить на апрув
- [ ] `recap` — итоги за период
- [ ] `inbox` — разобрать инбокс
- [ ] `onboard` — поднять доку по brownfield-проекту

Проверить наличие:
```bash
ls skills/
```

Если навык отсутствует — скопируйте его из актуального шаблона [github.com/kringova/artelush](https://github.com/kringova/artelush).
