# Канон документации проекта

Каждый проект на Артели обязан иметь набор документации — **лаконичный, но
обязательный**. Состав зависит от зрелости проекта (`status`): не требуем runbook
с проекта-идеи.

Документацией владеет **техписатель** (см. [roles.md](roles.md)); содержание берёт
у разработчика и аналитика как у SME. Тест существования любого документа — APC
(Audience · Purpose · Collaboration): нет ясных адресата и цели — документ не
пишется.

## Внешняя — Audience: потребитель продукта

Любой квалификации: обыватель, разработчик (для dev-инструментов), сам автор. Для
библиотеки/API внешняя дока техническая (API-reference, SDK) — но она внешняя,
потому что её читатель *использует продукт*, а не развивает проект.

| Документ | Когда обязателен |
|---|---|
| README / quickstart | всегда |
| Руководство пользователя / how-to | в проде |
| CHANGELOG / релизноутс | при первом релизе, далее ведётся через `close` |

## Внутренняя — Audience: контрибьютор проекта

Разработчик-человек или агент. Пишется **под машинное потребление** — открытый
`AGENTS.md`-формат, структурно, кратко. Это контракт проекта для любого агента:
вендор-независимость и экономия токенов.

| Документ | Когда обязателен |
|---|---|
| `AGENTS.md` (контракт проекта) | всегда |
| Архитектура (стек, модули, потоки) | всегда |
| **Запуск и деплой** (как поднять локально; **как именно катится в прод** — чем, куда, какие секреты, где CI-конфиг) | разработка+ |
| Решения (ADR-lite) | по мере роста |
| Runbook (как чинить в проде) | в проде |

Документ «Запуск и деплой» — точка истины для агента перед релизом (см.
[workflow.md](workflow.md) → «Перед деплоем — прочитай runbook»). Он обязан
соответствовать реальности: если процесс описан, но не работает (CI-конфиг не в
репо, секреты не заданы) — это вводящая в заблуждение дока, чинится в первую очередь.

## Вендор-агностичность конфига

Скиллы Артели уже переносимы (markdown, пути от корня репо). Та же независимость нужна
на уровне **конфига агента** — инструкции, permissions, hooks не должны жить только в
формате одного вендора. Три приёма, обязательные как конвенция:

1. **`AGENTS.md` — канонический контракт** в корне *каждого* репо проекта (не общая
   ссылка на артефакт Артели: файл нужен именно в корне, чтобы AGENTS-aware агенты —
   Codex, Cursor и др. — читали его нативно). Это источник правды для инструкций агента;
   общая методология Артели лежит в `_rules.md`, на неё `AGENTS.md` ссылается.
2. **`CLAUDE.md` = одна строка `@AGENTS.md`** — шим: Claude Code не читает `AGENTS.md`
   сам, но поддерживает `@`-импорт. Канон один, claude-only-исключения нет.
3. **Вендорская папка — read-only зеркало.** Конфиг правится только в каноне
   (`AGENTS.md` / общий источник), не в `.claude/` (`.codex/`, `.cursor/`).
   Сгенерированные вендорские mirror-папки — в `.gitignore`: в репо коммитится канон,
   не его проекции.

**Компилятор отложен.** Полноценный компилятор (`.agents/` → форматы вендоров одной
командой, как в `kltsv/ai-vendor-agnostic-setup`) вводится **только когда второй вендор
реально входит в ежедневную работу**. До этого триггера он — техдолг с хардкодом под
пару вендоров и риском «часть конфига молча не доезжает»; трёх приёмов выше достаточно.

**Применение:** конвенция обязательна для **новых** проектов; существующие дооснащаются
лениво — при ближайшем касании их конфига, без разовой миграции.

## Карта типов доки (APC)

Тип документа выводится из `Audience × Purpose`; наполнение — из `Collaboration` с
источниками (приоритет: артефакты → код → человек).

| Audience (роль в момент чтения) | Purpose | Документ | Collaboration (источник) |
|---|---|---|---|
| Потребитель | начать, быстрый успех | quickstart | разработчик |
| Потребитель | достичь цели | руководство / how-to | разработчик |
| Потребитель | понять, подходит ли | обзор «зачем это» | аналитик / владелец |
| Потребитель | что изменилось | changelog | разработчик + `close` |
| Контрибьютор | как здесь работать | `AGENTS.md` | разработчик |
| Контрибьютор | как устроено | архитектура | разработчик |
| Контрибьютор | почему так решили | решения (ADR-lite) | аналитик + разработчик |
| Контрибьютор | как починить | runbook | разработчик |

Авторский вклад относительно Diátaxis — агент как первоклассный читатель и ось
`Audience × Purpose`, а не «учусь/делаю». Подробно о письме — в
[styleguide.md](styleguide.md).

## Документация ведётся инкрементально

Значимое изменение → нужный документ обновляется **как часть DoD**, иначе задача не
уходит в `done`. Долг не копится. Релиз-гейт проверяет актуальность внешней доки и
changelog (см. [workflow.md](workflow.md)).

## Существующий проект → bootstrap

Проект без доки, переходящий на Артель, проходит режим `onboard`: канон собирается
из артефактов (ТЗ, заметки встреч, тикеты, старые доки) → кода → человека
(последним, для пробелов). Сгенерированное помечается «черновик, проверь».

## Место

Документация проекта живёт в репо проекта (`docs/`). Карточка проекта получает
ссылку и отметку «полнота документации под текущий статус».
