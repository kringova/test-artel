# FAQ

**Нужен ли Obsidian?**
Нет. Vault — обычный markdown, агент работает с файлами напрямую. Obsidian + Dataview дают красивые сводки (`_dashboard.md`, `tasks.md`) и кликабельные `[[ссылки]]`, но это надстройка для чтения, не требование.

**Есть ли веб-интерфейс без Obsidian?**
Да: `node dashboard/serve.js` (или `npm run dashboard`) запускает локальный веб-дашборд. Это полноценная альтернатива и дополнение к Obsidian — показывает проекты, канбан, беклог по RICE, страницы задач и поиск. Плюс два действия: быстрый захват в инбокс и апрув задач review→done прямо из браузера. Подробнее — [dashboard/README.md](../dashboard/README.md).

**Нужен ли GitHub / git?**
Тоже нет, но очень рекомендуется: история изменений, бэкап, несколько устройств. Если vault — git-репо, агент сам коммитит и пушит после правок (так написано в правилах).

**Шаблон публичный — а мои данные?**
Ваш vault — отдельный репозиторий, делайте его приватным. В шаблоне ваших данных нет и не будет; связь между шаблоном и вашим vault не автоматическая.

**Вышло обновление шаблона — как подтянуть?**
Вручную и выборочно: обновления методологии — это редкие правки в `_rules.md` и `skills/`. Посмотрите diff в шаблоне и перенесите нужное. Автосинк сознательно не предусмотрен: живой vault важнее свежести конвенций, и ничто не должно уметь его перезаписывать.

Перенос обновлений шаблона оформляй задачей с DoD: (1) шаблонные файлы перенесены; (2) пользовательские данные (`projects/`, `_inbox/`) не перезаписаны; (3) локальный дашборд запускается без ошибок. Заводи такую задачу в карточке самого vault-проекта (инфра-работа над vault), а не во вложенном пользовательском проекте.

**Могу я вести задачи на английском / поменять конвенции?**
Да. Правила — ваши файлы: поменяйте язык, шкалы RICE, статусы в `_rules.md` и навыках. Агент следует тому, что написано в vault, а не «зашитому» поведению.

**Что делает «отслеживаемость» на практике?**
Агент не имеет права молча сделать работу: код написан → задача `done`; обсудили фичу → задача `todo`; выяснилось, что уже сделано → задача `done` постфактум. Через месяц вы (или новый агент) открываете карточку проекта и за минуту видите всю историю.

**Несколько устройств конфликтуют?**
Практически нет: записи инбокса — отдельные файлы, задачи правятся по одной. Правило одно: `git pull --rebase --autostash` в начале работы (навык `inbox` делает это сам).

**Как кидать мысли в инбокс с телефона?**
Любой способ создать markdown-файл в `_inbox/` подойдёт: iOS Shortcuts c коммитом через GitHub API, веб-форма на своём сервере, телеграм-бот, GitHub mobile (Add file). Формат записи — три строки frontmatter + текст, см. [reference.md](reference.md).

**Работает ли с другими агентами, не Claude Code?**
Конвенции — да: `AGENTS.md` (вендор-нейтральный канонический файл; Claude Code читает его через шим `CLAUDE.md` → `@AGENTS.md`) и `_rules.md` читаемы любой моделью с доступом к файлам. Навыки `skills/` — формат Claude Code, но это обычный markdown с пошаговыми инструкциями: скормите их любому агенту как промпт.

**Чем это отличается от Jira/Linear/Todoist?**
Тем, что трекер ведёт агент, а не вы; данные — plain markdown в вашем git; и трекер совмещён с памятью агента о проекте (контекст, решения, лог сессий), а не только списком задач. Минус честный: нет мультипользовательности — это персональная система.

**Сколько проектов тянет один vault?**
Десяток активных — комфортно. Сводка `sync` и сквозные id не деградируют; узкое место — скорее ваше внимание, и RICE как раз помогает.

**Что добавляют роли и APC поверх беклога?**
Беклог решает, *что* делать и в каком порядке. Роли (аналитик, разработчик, ревьюер, техписатель) подключаются по важности задачи и следят за качеством и документацией; документация ведётся по авторской модели APC (Audience · Purpose · Collaboration). Подробно — в [roles.md](roles.md), [workflow.md](workflow.md), [doc-canon.md](doc-canon.md). Работают через чтение этих файлов — настройки и конкретного вендора не требуют.
